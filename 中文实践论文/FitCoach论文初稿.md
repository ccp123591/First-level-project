## 1 引言

随着人口老龄化程度的不断加深，老年群体对于居家康养和日常锻炼的需求持续增长。世界卫生组织发布的身体活动指南指出，规律的身体活动能够显著改善老年人的心肺功能、肌肉力量、平衡能力和整体健康状态，是促进健康老龄化的重要方式[1]。相关综述研究进一步表明，面向老年人的运动干预能够在肌力、平衡、功能活动能力等方面产生积极效果，并有助于提升生活质量[2]。同时，针对社区居住老年人的系统综述显示，科学设计的运动训练能够有效降低跌倒发生率[3]。因此，如何为老年人提供安全、便捷、可持续的居家训练支持，已成为智能健康应用值得关注的研究方向。

然而，在真实居家场景中，老年用户往往面临动作理解不充分、训练节奏把握困难、缺少现场指导以及对复杂数字产品操作不熟悉等问题。现有常见的健身视频平台或运动打卡类应用大多以内容展示和结果记录为主，虽然能够提供基础训练资源，但通常缺乏对训练过程的实时识别、动作纠正和即时反馈能力，难以满足老年用户对“低门槛、强引导、重安全”的使用需求。另一方面，已有研究表明，借助数字健康手段开展居家运动干预，能够改善老年人的下肢力量、功能能力和健康相关生活质量，并在一定程度上减少跌倒事件发生[4]。这说明，将数字技术与老年居家锻炼需求相结合具有较好的应用基础，但系统设计必须进一步兼顾适老化、可解释性与弱网环境下的可用性。

近年来，人体姿态识别技术与浏览器端计算能力的快速发展，为构建轻量化智能健身陪练系统提供了新的技术路径。MediaPipe 是一种面向跨平台感知任务的工程化处理框架，能够在浏览器、移动端等环境中对视频流进行实时处理，并支持人体关键点检测等任务[5]；BlazePose 则是在该技术体系下提出的轻量化人体姿态估计方案，重点面向移动端和实时场景优化，能够为普通摄像头条件下的动作识别提供较稳定的关键点输出[6]。基于上述技术，可以在浏览器端直接完成姿态检测、动作计数与结果反馈，不需要用户额外购置专用硬件，也不必将训练视频上传到云端，从而在部署成本、响应速度和隐私保护方面形成一定优势。考虑到家庭网络可能存在弱网、断网等情况，若系统能够在本地完成训练记录保存和基础反馈，则更符合老年用户连续使用的实际需求。

基于此，本文以一级项目 FitCoach 为研究对象，采用系统设计与工程实现相结合的方法，设计并实现一个面向居家老年人的智能健身陪练系统。该系统以前端姿态识别为核心，结合动作状态机判定、多维评分机制、语音提示、IndexedDB 本地存储和 PWA 离线能力，形成较完整的训练辅助闭环。本文的主要工作体现在以下三个方面：第一，将浏览器端姿态识别技术应用于老年居家锻炼场景，构建无需上传视频的本地训练识别链路；第二，采用离线优先的数据保存机制，提升系统在家庭弱网环境下的连续使用能力；第三，基于规则驱动方式实现节奏、稳定性、深度、对称性和完成率等多维可解释评分，并结合语音反馈增强训练结果的直观性与可理解性。本文后续内容安排如下：第 2 节介绍系统实现所涉及的相关技术，并结合老年居家锻炼场景对系统需求进行分析；第 3 节从总体架构、模块划分、数据流设计和前后端协同等角度对 FitCoach 一级项目的整体方案进行阐述；第 4 节重点介绍姿态识别、动作判定、多维评分、本地记录管理和后端支撑接口等核心模块的设计与实现；第 5 节结合系统运行结果，从功能验证和典型训练结果两个方面对系统进行分析，并总结当前实现中的不足；第 6 节对全文进行总结，并对后续的适老化优化与系统扩展方向进行展望。

## 2 相关技术与系统需求分析

### 2.1 相关技术基础

FitCoach 面向的是居家老年用户，因此系统技术选型不仅要满足动作识别的基本功能，还需要兼顾低部署门槛、较好的实时性、弱网可用性以及较低的操作复杂度。结合项目当前实现情况，本文主要采用浏览器端人体姿态识别、渐进式 Web 应用（Progressive Web App，PWA）、浏览器本地结构化数据库（IndexedDB）以及 Vue 3 与 Spring Boot 前后端分离架构作为系统的核心技术基础。

首先，在姿态识别技术方面，系统采用 MediaPipe Pose 作为浏览器端人体关键点检测的核心方案。该技术能够从视频流中提取人体关节关键点位置，并为后续动作判定、关节角度计算和训练计数提供基础数据支持[5]。与依赖云端视频分析或专用硬件传感器的方案相比，浏览器端姿态识别可以直接调用普通摄像头完成训练过程感知，从而降低设备购置成本和使用门槛。对于居家老年用户而言，这种“打开浏览器即可训练”的方式更加符合其日常使用习惯。与此同时，BlazePose 作为面向实时场景优化的人体姿态估计方案，能够在普通摄像头条件下提供较稳定的关键点输出[6]，为本项目中的深蹲、前屈、弓步蹲、臀桥和平板支撑等动作识别提供了可行支撑。

其次，在应用形态方面，系统采用渐进式 Web 应用技术。所谓 PWA，是指兼具网页访问便捷性和部分原生应用体验特征的 Web 应用形态，其典型特点包括可安装、可离线访问以及支持资源缓存等能力。FitCoach 前端通过注册服务工作线程（Service Worker）实现静态资源缓存，并结合应用清单文件（manifest.json）提供安装入口，使用户能够以接近应用程序的方式访问训练系统。对于居家老年用户而言，PWA 形态的优势在于无需经过复杂安装流程即可使用，同时在网络不稳定时仍可打开系统并进入基础训练页面，这对于提升系统的可达性和持续使用体验具有积极意义。

再次，在数据管理方面，系统采用 IndexedDB 作为浏览器端本地训练记录存储机制。IndexedDB 是浏览器提供的一种面向结构化数据的本地持久化存储方式，适合保存对象化、记录型的数据内容。FitCoach 将每次训练结束后的动作名称、次数、评分、时长和训练时间等信息优先写入本地数据库，再在满足条件时尝试上传后端。该设计有两个直接优点：一是能够避免因网络中断导致训练结果丢失；二是使用户在游客模式或未登录状态下也能够完成本地训练记录管理。对于老年居家锻炼场景而言，这种离线优先的数据处理思路比完全依赖在线接口提交的方式更具稳定性和实用性。

最后，在系统架构方面，项目采用 Vue 3 与 Spring Boot 的前后端分离实现方式。Vue 3 主要负责前端页面展示、训练交互、状态管理和浏览器端业务逻辑组织，Spring Boot 则负责用户认证、训练记录、排行榜、计划管理等后端接口支撑。前端通过统一封装的 Axios 请求实例与后端进行通信，后端接口统一采用 `/api/` 前缀和标准响应结构返回数据。这种分层方式一方面便于将姿态识别、训练交互与后端业务能力解耦，另一方面也为后续系统扩展提供了清晰的结构基础。虽然当前部分扩展业务模块仍处于演示态或骨架态，但从总体设计上看，这一技术组合能够较好支撑本文所面向的居家老年智能健身陪练应用场景。

### 2.2 系统需求分析

结合老年人居家训练的实际特点以及 FitCoach 一级项目当前的实现目标，本文将系统需求划分为功能需求和非功能需求两个层面。功能需求关注系统“需要做什么”，非功能需求则关注系统“应当以何种方式工作”，特别是是否适合老年用户长期、稳定地使用。

在功能需求方面，系统首先需要具备训练动作选择与训练目标设置能力。由于不同老年用户的身体状况、训练习惯和运动能力存在差异，系统应支持用户根据自身需要选择不同动作，并设置目标次数或训练节奏。其次，系统需要具备摄像头采集与姿态识别能力，即能够在训练过程中实时采集人体动作信息，并提取可用于动作判定的关键点数据。再次，系统需要具备动作计数与状态提示能力，能够根据动作变化过程判断训练是否到位、是否完成一次有效动作，以及在必要时给出纠错提示。除此之外，系统还需要具备训练结果评分与训练报告输出能力，用于向用户展示节奏、稳定性、深度、对称性和完成率等维度的训练结果，从而帮助用户更直观地理解训练表现。最后，系统还应具备本地记录保存、历史记录查看、基础后端接口支撑以及语音反馈辅助等功能，以构成较完整的训练闭环。

在非功能需求方面，首先需要考虑实时性。动作识别与计数过程如果存在较明显延迟，将直接影响训练反馈的连贯性和用户体验，因此系统应尽量保证关键点检测、角度计算和状态判断在训练过程中能够及时完成。其次需要考虑易用性，尤其是老年用户的使用门槛问题。系统界面应尽量简洁，功能入口应清晰明确，操作流程不宜过长，以减少学习成本和误操作概率。第三，需要考虑反馈结果的可解释性。单纯给出一个综合分数往往难以帮助用户理解训练问题，因此系统还应通过分维度评分和语音提示等方式增强结果说明能力。第四，需要考虑弱网与离线可用性。由于家庭网络环境并不总是稳定，系统应尽量避免把训练主流程完全建立在网络持续可用的前提之上。第五，需要考虑隐私保护与可扩展性。训练视频和人体姿态数据涉及个人隐私信息，本地处理策略有助于减少隐私暴露风险；同时，系统还应保留与后端服务协同的扩展空间，以便后续接入更完整的用户管理、训练计划和智能反馈能力。

对于面向居家老年人的 FitCoach 而言，上述功能需求与非功能需求并不是彼此独立的。低门槛使用、实时动作识别、语音引导、本地记录保存以及离线优先等要求共同构成了系统设计的基本约束条件。也正是在这些约束下，本文选择以浏览器端姿态识别和轻量化 Web 应用形态为主线，形成后续总体设计与核心模块实现的技术基础。
