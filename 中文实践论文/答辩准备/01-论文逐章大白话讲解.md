## 起步：一句话讲清楚你做了什么

> 你做了一个**网页**，老年人**打开浏览器**就能用。系统**用普通摄像头**看他练动作，**自动数次数 + 打分**，**不联网也能存记录**，登录后再上传到云端。

如果有人问「这是 App 吗」，标准回答：

> 这是**一个网页应用，但用了 PWA 技术包装成"像 App"**——可以加到桌面、不联网也能打开。比真正的 App 不需要从应用商店下载安装。

## 第 1 章 · 引言（P1–P7）

引言一共 4 段，我把每段在干什么用一句话说明白。

### 1.1 第 1 段（P1）：为什么要做老年人居家健身

**论文核心句**：
- 人口老龄化越来越严重，老年人居家锻炼需求增长
- 世卫组织（WHO）2020 年的指南说：规律运动能改善老人心肺、肌肉、平衡 [1]
- 综述研究[2] 说：老年人运动干预对肌力/平衡有明显效果
- 综述研究[3] 说：科学训练能降低跌倒发生率
- 所以做一套老人能在家里安全用的训练系统是有意义的

**大白话**：
> 老人越来越多，他们在家想锻炼，但乱练容易摔。WHO 都说运动好，所以做这个有理论支撑。

**如果被追问"为什么针对老年人"**：
> 三点：① 国家老龄化趋势是真问题；② 老年人在家锻炼最容易出现"动作不对、节奏不稳、跌倒"的安全风险；③ 现有产品对他们不友好——下面第 2 段就讲这个。

### 1.2 第 2 段（P2）：现有产品有什么毛病

**论文核心句**：
- 老年用户普遍存在：动作理解不到位、节奏把握困难、缺现场指导、对复杂数字产品不熟
- 现有的健身视频/打卡 App 大多只是"放视频 + 记录次数"，不能实时纠错
- 已有研究[4] 说：数字健康手段做居家运动干预对老年人确实有效
- 所以**有应用基础**，但要做适老化、可解释、弱网可用

**大白话**：
> 老人不会用复杂 App。市面上的健身 App 只能给你看视频、点几下打卡，但你做对了没？没人告诉你。我们要做的就是**把"实时纠错"这一步加上**。

### 1.3 第 3 段（P6）：技术上为什么现在能做了

**论文核心句**：
- 这几年人体姿态识别技术 + 浏览器端算力都涨了
- **MediaPipe** 是谷歌做的跨平台感知框架，能在浏览器/手机里实时处理视频[5]
- **BlazePose** 是 MediaPipe 体系下的轻量姿态估计模型，专门给移动端做的[6]
- 这些技术让"浏览器里直接做姿态识别"成为可能 → 不需要额外硬件、不用上传视频
- 家里网经常不稳，所以系统得能离线用

**大白话**：
> 以前做姿态识别要么得有专业摄像头，要么得把视频传到服务器算。现在 Google 出了 MediaPipe + BlazePose 这套技术，**直接在你家的 Chrome 浏览器里就能跑**。视频不出你家，省钱又安全。

**术语速查**：
- **MediaPipe**：Google 2019 年开源的"感知任务流水线"框架。可以理解为"一套让 AI 在浏览器/手机里跑视觉任务的工具箱"。
- **BlazePose**：MediaPipe 工具箱里的一个具体模型，专门检测人体 33 个关键点（鼻子、肩、肘、手腕、髋、膝、踝……）。论文表 2 里写的"23/25/27"就是 BlazePose 给左髋/左膝/左踝标的编号。
- **姿态识别（Pose Estimation）**：从视频/图片里找出人体的关键点位置。

### 1.4 第 4 段（P7）：本文做了什么 + 三个创新点

**论文里写的三个创新点（必背）**：

| # | 创新点 | 说人话 |
|---|---|---|
| 1 | 把浏览器端姿态识别用到老年居家场景，**无需上传视频**的本地训练识别链路 | 视频不出你家，老人不用学怎么传视频 |
| 2 | **离线优先**的数据保存机制，弱网/断网也能继续用 | 哪怕没网也能练 + 存记录 |
| 3 | 基于规则的**多维可解释评分**（节奏/稳定/深度/对称/完成率）+ 语音反馈 | 不只给 90 分，还告诉你"哪一项扣分了" |

**如果被追问"创新点是不是太弱"**：
> 这是**系统设计与实现类**的课程论文，不是算法原创论文。论文 P81 已经写明："本文的主要工作并不在于提出全新的姿态估计算法，而在于结合实际应用需求，将前端识别、训练反馈和离线优先机制**组织成可运行的系统方案**"。我的创新点是**场景适配 + 工程组织 + 可解释设计**这三层。

## 第 2 章 · 相关技术与系统需求分析

### 2.1 相关技术基础（P10–P14）

论文按四块讲技术选型，我逐个解释。

#### ① MediaPipe Pose / BlazePose（姿态识别）

**论文表述**：用 MediaPipe Pose + BlazePose 在浏览器端检测人体关键点。

**它是怎么工作的**：
1. 你打开摄像头
2. JavaScript 把视频每一帧送给 MediaPipe
3. MediaPipe 调用 BlazePose 模型推理，吐回 33 个关键点的 (x, y, 可见度) 坐标
4. 你的代码再用这 33 个点做后续计算

**为什么用它**：
- 在浏览器里跑（不用装专门的软件）
- 用 WebAssembly + GPU 加速（速度够实时）
- 模型是 Google 训好的，开箱即用（你不用自己训练）

**如果被追问"33 个关键点是怎么编号的"**：
> BlazePose 的标准编号 0~32，0 是鼻子，11/12 是左/右肩，13/14 是左/右肘，23/24 是左/右髋，25/26 是左/右膝，27/28 是左/右踝。论文表 2 里"髋-膝-踝"对应的就是 23-25-27（左侧）和 24-26-28（右侧）。

#### ② PWA（渐进式 Web 应用）

**论文表述**：通过 Service Worker 缓存静态资源，配合 manifest.json 提供安装入口，弱网下也能基本可用。

**大白话**：
> PWA 就是**让网页变得像 App**——可以加到手机桌面、能离线打开。靠两个技术：
> - **Service Worker**（服务工作线程）：浏览器后台里跑的小程序，把网页的 HTML/CSS/JS/图标都缓存到本地，下次打开不用重新下载
> - **manifest.json**（应用清单）：一个配置文件，告诉浏览器"这个网页可以装到桌面，图标长这样"

**为什么对老年人重要**：
- 不用进应用商店搜+下载（步骤少）
- 第一次打开后，第二次不联网也能打开
- 占空间小（不是真 App）

#### ③ IndexedDB（本地数据库）

**论文表述**：训练记录优先写入 IndexedDB，再在登录后上传后端。

**大白话**：
> 浏览器里**自带的小型数据库**。你写的代码可以往这里存对象（动作名、次数、评分、时长……），下次打开网页还能查得到。
> 跟 LocalStorage 的区别：LocalStorage 只能存字符串、容量小（5MB），IndexedDB 能存结构化对象、容量大（几百 MB～GB）。

**为什么用它**：
- 老人家网络不稳，先存本地不会丢
- 游客模式（没登录）也能存
- 离线场景的兜底

**如果被追问"为什么不直接传后端"**：
> 因为家里网络可能不稳定。如果训练做完一上传就失败，那这次成绩就丢了。先存本地再"有空"上传，是**离线优先**思路。这就是创新点 ②。

#### ④ Vue 3 + Spring Boot（前后端架构）

**论文表述**：前端 Vue 3 负责页面交互、训练流程；后端 Spring Boot 负责认证、记录、扩展接口。

**大白话**：
> - **Vue 3**：写前端页面的 JavaScript 框架。可以理解为"组件化造网页的工具"。
> - **Spring Boot**：写后端服务的 Java 框架。负责接收前端请求、跟数据库打交道、返回数据。
> - **前后端分离**：前端只管显示和交互，后端只管数据。两边通过 HTTP 接口（论文里说的 `/api/`）通信。这样改前端不用动后端，反之亦然。

## 第 3 章 · 总体设计

### 3.1 系统总体架构（P22–P28）

**论文核心**：三层架构。

```
┌─────────────────────────────────────────┐
│ 前端训练层（浏览器跑）                  │
│   摄像头 → 姿态识别 → 计数 → 评分 → 报告 │
└─────────────────────────────────────────┘
                  ↕ HTTP /api/...
┌─────────────────────────────────────────┐
│ 后端服务层（Spring Boot）               │
│   认证 / 记录管理 / 扩展业务接口        │
└─────────────────────────────────────────┘
                  ↕
┌─────────────────────────────────────────┐
│ 数据存储层                              │
│   浏览器 IndexedDB  +  服务端数据库     │
└─────────────────────────────────────────┘
```

**为什么这样分（被追问就答）**：
- **训练核心放浏览器端**：用户家里没网也能练，训练流程不依赖网络
- **后端不介入每次识别**：每次摄像头识别都不需要联网请求服务器，否则会卡
- **存储双层**：本地保证不丢、云端保证长期管理

**对应论文图 2**：你要说"图 2 展示了这个三层架构"。

### 3.2 功能模块划分（P30–P31）

论文把项目分**四个模块**：

1. **训练模块** —— 核心
   - 动作选择 / 姿态识别 / 动作判定 / 计数 / 评分
2. **本地记录模块**
   - 保存到 IndexedDB / 历史查看 / CSV 导出
3. **设置与反馈模块**
   - 语音播报 / 节拍参数 / 自动暂停 / 训练报告展示
4. **后端支撑模块**
   - 用户认证 / 训练记录接口 / 扩展业务（计划/排行榜/动态）

**陷阱提醒**：第 4 块"后端支撑"被问到时，要说"它是项目整体的扩展能力支撑，**不是本论文的核心**"——这是论文 P82 反复强调的措辞。

### 3.3 数据流与业务流程（P33–P34）

**主链路**（必背，对应论文图 3）：

```
选动作 → 启动姿态识别 → 摄像头采帧 → 关键点输入判定逻辑
   → 角度计算 + 状态判断 + 计数
   → 训练结束 → 生成评分 + 报告
   → 优先写入本地（IndexedDB）
   → 登录条件下尝试上传后端
```

**关键设计：本地优先**
> 训练结果**先写本地，再上传**。即使网络断了，训练记录也不会丢。这是论文创新点 ②。

### 3.4 数据存储与前后端协同（P38–P39）

**论文核心**：双层存储 + 前端跑核心训练 + 后端做长期管理。

| 数据类型 | 存哪里 | 为什么 |
|---|---|---|
| 单次训练记录（动作/次数/评分/时长） | 浏览器 IndexedDB | 离线可用、游客也能存 |
| 训练参数（语音/节拍/主题） | 浏览器 LocalStorage | 配置项小，简单存 |
| 用户账号 / 长期累计记录 / 计划 / 排行榜 | 后端数据库（H2 / MySQL） | 多端同步、长期管理 |

**坦白话**：论文 P59 已经写"自动补传机制尚未完全闭环"——这是已知不足，被问到承认即可。

## 第 4 章 · 核心模块设计与实现（最重点）

### 4.1 浏览器端姿态识别模块（P42–P43）

**做了什么**：
1. 训练页一打开，初始化 MediaPipe Pose 模型
2. 调用摄像头，把每一帧塞给 MediaPipe
3. MediaPipe 返回 33 个关键点
4. 根据当前动作（深蹲/前屈/俯卧撑/弓步蹲/臀桥/平板/开合跳），**只看相关的关键关节**

**为什么"只看相关关键关节"**（这是个加分点）：
> 深蹲跟下肢有关，所以只关心**髋（23/24）-膝（25/26）-踝（27/28）**；
> 前屈跟脊柱有关，所以只关心**肩（11/12）-髋（23/24）-膝（25/26）**。
> 这样既减少计算量，又让动作判定逻辑更聚焦。

**论文 P43 的态度**：
> "该模块的重点不在于复杂模型创新，而在于以较低门槛支撑稳定、实时的训练识别过程。"
> ——你不是要发明新模型，是要把现成模型用得稳。

### 4.2 动作判定与计数模块（P45–P49 + 表 2 + 公式 1）

**核心思路三步走**：

**第一步：算关节角**
> 拿三个关键点 a、b、c（比如髋-膝-踝），b 是顶点，算出在 b 处的夹角。这就是论文公式 (1)。
> 公式细节看 `02-两个公式怎么讲.md`，这里只要记住一句：
> **「用 atan2 算两个向量的夹角，比 arccos 数值上更稳定。」**

**第二步：状态机计数**

每次动作有 3 个状态：

```
              角度 < 阈值下限（例：90°）
   ┌────────────────────────────────────┐
   │                                    ▼
[idle / up] ───────────────→ [down]  
   ▲                                    │
   │                                    │
   └─────── 角度 > 阈值上限（例：160°）─┘
              （此时 reps += 1）
```

人话翻译：
- 初始状态 `idle`，膝盖伸直时角度 ≈ 180°
- 蹲下去，角度变小，**当角度跌破 90°** → 进入 `down` 状态
- 站起来，角度变大，**当角度超过 160°** → 回到 `up` 状态，**这时计数 +1**

**为什么用状态机不是"每次都计数"**：
> 否则你蹲下去那一瞬间会狂加好几次。状态机保证"必须先下、再上才算一次"，符合人对"一次动作"的直觉。

**第三步：阈值表（论文表 2）**

| 动作 | 关键点 | 阈值下限 | 阈值上限 | 类型 |
|---|---|---|---|---|
| 深蹲 | 髋-膝-踝 | 90° | 160° | 计数型 |
| 前屈伸展 | 肩-髋-膝 | 60° | 160° | 计数型 |
| 俯卧撑 | 肩-肘-腕 | 80° | 160° | 计数型 |
| 弓步蹲 | 髋-膝-踝 | 100° | 170° | 计数型 |
| 臀桥 | 肩-髋-膝 | 150° | 175° | 计数型 |
| 平板支撑 | 肩-髋-踝 | — | — | 时间型 |
| 开合跳 | 肩-肘-腕 | — | — | 计数型 |

**关键点：**
- 阈值是**经验值**，根据动作的运动学常识设定（深蹲蹲到 90° 算到位）
- 平板支撑没阈值因为是**计时型**，不是计次型
- 你被问到"阈值哪来的"就说："基于动作运动学经验设定，并通过实测调整"

**还有一个"平滑处理"细节**（论文 P49）：
> 摄像头帧间会抖（识别噪声）。系统对角度序列做**滑动窗口平均**（最近 5 帧的平均值），避免因为一帧抖动就误判一次动作。

### 4.3 多维评分模块（P53–P56 + 公式 2）

**核心思路**：不只给一个总分，给 5 个维度分。

| 维度 | 含义 | 怎么算（一句话） |
|---|---|---|
| 节奏分 R | 动作节奏稳不稳 | 两次动作时间间隔 vs 目标节拍（BPM）的偏差 |
| 稳定分 T | 动作底部稳不稳 | 在最低位置附近的角度方差 |
| 深度分 D | 蹲得够不够深 | 实际最低角度 vs 理想角度的偏差 |
| 对称分 Y | 左右对称吗 | 左侧角度 vs 右侧角度的差值 |
| 完成率 C | 完成多少次 | 实际次数 / 目标次数 |

**综合分公式**（论文公式 2）：

> **S = 0.25R + 0.25T + 0.20D + 0.15Y + 0.15C**

权重逻辑：
- 节奏 + 稳定 各 25%（最重要 —— 老人摔倒往往是动作不稳）
- 深度 20%（重要，但比稳定次一级）
- 对称 + 完成率 各 15%（辅助）

**详细的算法和被追问怎么答**：看 `02-两个公式怎么讲.md`。

**为什么这么设计**（适老化的体现）：
> 比起只给一个 90 分，5 维评分让老人能看懂"我哪里扣分了"。
> 比如发现"深度只有 60 分"，就知道下次要蹲深一点。**这就是论文反复强调的"可解释性"。**

### 4.4 本地记录与后端支撑（P58–P59）

**前端这边干了什么**：
1. 训练结束 → 把结果（动作名、次数、评分、时长、5 维分）打包成对象
2. 写到 IndexedDB 的 `sessions` 表
3. 标记 `synced = 0`（未同步）
4. 如果用户登录了 + 网络通 → 调 `/api/sessions` 上传 → 上传成功后改 `synced = 1`

**后端这边提供了什么接口**：
- `/api/auth/...` 登录/注册/Token 刷新
- `/api/sessions` 训练记录创建/查询/批量上传
- `/api/exercises` 动作库
- `/api/plans` 训练计划
- `/api/leaderboard/...` 排行榜
- `/api/social/...` 动态 Feed
- `/api/coach/feedback` AI 教练反馈
- `/api/admin/...` 管理后台

**关键坦白**（P59 论文原话）：
> "当前项目已经形成'本地优先保存 + 登录后条件上传'的基本链路，但**自动补传机制尚未完全闭环**。"
> 翻译：手动触发上传是通的，但"网络一恢复自动把没上传的全部传上去"还没做。被问到要承认。

### 5.3 典型训练结果分析（P71–P72 + 图 5）

**论文用深蹲做例子**，你要会说：

> "以深蹲为例，系统能输出次数、时长、综合评分，并进一步给出 5 个维度的子分。比起只记次数的传统打卡 App，这种方式让用户更直观地理解训练表现。"

**5 维评分怎么帮老人**（论文 P72 的关键论点）：
- 用户**深度不够** → 深度维度分数低 → 下次知道要蹲深
- 用户**左右偏差大** → 对称性维度分数低 → 下次知道要平衡用力
- ——**让用户知道问题大致出在哪，而不只看到一个抽象总分**

**重要坦白**（图 5 处的 P72）：
> "图 5 中所示评分数据为界面功能演示，**当前后端评分接口尚未完全接通**，实际评分数值将在系统完善后进一步验证。"
> 这段话是论文事先做好的"防御"——评委挑剔图 5 的数据怎么来的，你直接引用这一句即可。

### 5.4 当前不足（P77，必背）

**4 项不足**（顺序固定）：

1. **摄像头敏感度** —— 光线暗、遮挡、摄像头角度不对时识别会不稳
2. **规则驱动** —— 评分本质上是工程化规则，不能替代专业康复指导
3. **自动同步未闭环** —— 本地存能存，但网络恢复后自动补传没做
4. **缺老年用户实测** —— 还没做大规模适老化实验

**作答模板**：
> "我们在论文 5.4 节如实承认了 4 个不足：第一是摄像头敏感性，第二是规则驱动评分的局限，第三是自动同步链路，第四是缺乏真实老年用户的可用性实验。这些都列入了第 6 章的展望中后续工作。"
